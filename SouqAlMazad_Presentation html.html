Key Tables in Detail

product — Marketplace Catalog

sale_type ENUM('direct','auction','both') drives dispatch. create_auction validates sale_type IN ('auction','both') before insert. stock_quantity, view_count, is_active soft-delete.

auction — Bidding Engine

current_highest cached aggregate updated by place_bid. winner_buyer_id uses ON DELETE SET NULL (preserves history). Status: upcoming → active → ended → cancelled.

order — 4-Table Composite

7-status lifecycle. Stores total_amount, discount_amount, final_amount (deliberate 3NF violation for audit). Children: order_item, payment, shipment — all CASCADE on order_id.

review — Verified Purchase

3-FK design: product_id + buyer_id + order_id. add_review validates buyer actually bought the product via that order. CHECK(rating BETWEEN 1 AND 5).

Normalization: 1NF → 2NF → 3NF ✅

1NF — Every column atomic. No repeating groups (product_image is separate table).

2NF — No partial dependencies. All tables use single-column surrogate PK.

3NF — No transitive deps. store_name in store, not in product.

3 Deliberate Denormalizations: auction.current_highest (avoid MAX on every bid check) · auction.winner_buyer_id (avoid re-scanning bids) · store.rating_avg (dashboard performance)

SQL Queries — JOINs (Q1–Q6)

Q3 Order Master View
SELECT o.order_id, u.first_name, p.title, oi.subtotal, pay.status, sh.status FROM `order` o INNER JOIN buyer bu ON o.buyer_id = bu.buyer_id INNER JOIN `user` u ON bu.user_id = u.user_id INNER JOIN order_item oi ON o.order_id = oi.order_id LEFT JOIN payment pay ON o.order_id = pay.order_id LEFT JOIN shipment sh ON o.order_id = sh.order_id;

🔑 INNER on buyer/order_item (always exist). LEFT on payment/shipment — new orders without payment aren't silently dropped. This join-type choice is what makes the query correct.

Q5 Sellers with Product Count
SELECT u.first_name, u.last_name, s.store_name, COUNT(p.product_id) AS products FROM seller sl INNER JOIN `user` u ON sl.user_id = u.user_id INNER JOIN store s ON sl.seller_id = s.seller_id LEFT JOIN product p ON s.store_id = p.store_id GROUP BY u.first_name, u.last_name, s.store_name;

🔑 LEFT JOIN product — empty stores still appear with COUNT=0. ISA traversal: seller → user to get the name. Classic "count children per parent" pattern.

SQL Queries — GROUP BY (Q7–Q14)

Q8 Avg Rating with HAVING
SELECT p.title, AVG(r.rating) AS avg_rating, COUNT(r.review_id) AS num_reviews FROM product p LEFT JOIN review r ON p.product_id = r.product_id GROUP BY p.title HAVING COUNT(r.review_id) > 0;

🔑 HAVING not WHERE — the filter is on an aggregate (COUNT). WHERE runs before grouping, HAVING runs after. Textbook idiom for filtering aggregates.

Q9 Revenue per Store
SELECT s.store_name, SUM(oi.subtotal) AS total_sales FROM store s INNER JOIN order_item oi ON s.store_id = oi.store_id INNER JOIN `order` o ON oi.order_id = o.order_id WHERE o.status != 'cancelled' GROUP BY s.store_name ORDER BY total_sales DESC;

🔑 Revenue attributed via order_item.store_id, not order — because one order can span multiple stores. WHERE filters cancelled orders before aggregation.

SQL Queries — Subqueries (Q15–Q20)

Q13 Correlated Subquery
SELECT p.title, p.price, c.name FROM product p INNER JOIN category c ON p.category_id = c.category_id WHERE p.price > ( SELECT AVG(p2.price) FROM product p2 WHERE p2.category_id = p.category_id -- correlated! );

🔑 Inner query references outer p.category_id — AVG recomputed per row. Products priced above their own category average. Cannot be written as a single GROUP BY.

Q20 Two-Level Nested
SELECT u.first_name, spending.total FROM ( SELECT buyer_id, SUM(final_amount) AS total FROM `order` WHERE status!='cancelled' GROUP BY buyer_id ) AS spending JOIN buyer bu ON spending.buyer_id = bu.buyer_id JOIN `user` u ON bu.user_id = u.user_id WHERE spending.total > (SELECT AVG(t.total) FROM (...) t);

🔑 Most complex query — derived table + nested aggregate. Same GROUP BY appears twice. In production: rewrite as CTE (WITH spending AS ...) to compute once.

Q17 — NOT IN anti-join: stores with zero sales. Works because store_id is NOT NULL (no NULL-in-NOT-IN gotcha).

Q18 — NOT EXISTS preferred over NOT IN when NULLs possible. Optimizer can short-circuit on first match.

Q19 — Scalar subqueries in SELECT: MAX + COUNT per auction without GROUP BY. Equivalent to cached current_highest.

⚠️ Without Locking

Two buyers read the same current_highest simultaneously. Both pass validation. Both INSERT. One bid overwrites the other — silent data corruption.

FOR UPDATE prevents this entirely.

Same pattern in:
direct_buy_product · close_auction_and_create_order

If any step raises an exception → ROLLBACK → database returns to previous state. No partial orders, no phantom bids.
